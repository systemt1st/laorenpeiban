## 1. 文档定位与边界声明

本文档描述的是 **目标架构（To-Be）**，用于指导“老人陪伴助手”从当前版本向下一代系统演进。  
`GoldenAge Guardian（金色年华守护者）` 是内部架构代号，对外产品名称保持“老人陪伴助手”不变。

1. 本文不宣称仓库当前已全部实现文中能力。

## 2. 项目愿景与架构目标

### 2.1 项目愿景

围绕“陪伴 + 安全 + 可追溯”三条主线，打造老年人可长期信任的 AI 陪伴系统：

1. 日常可依赖：语音优先、低门槛交互，稳定提供情绪陪伴与生活辅助。
2. 风险可感知：对健康风险表达进行多级识别、确认和升级响应。
3. 家属可协同：关键事件可追踪、可解释，形成家庭共同照护闭环。

### 2.2 架构目标

1. 高可用：核心对话与风险识别链路在单点故障下可降级运行。
2. 高可控：Agent 行为可观测、可审计、可回放。
3. 高安全：默认最小权限、工具沙箱隔离、敏感数据分级保护。
4. 高演进性：通过 Supervisor + 核心业务 Agent 模式支持后续能力扩展。

## 3. 总体系统层级

GoldenAge Guardian 采用四层架构：**感知层 / 编排层 / 执行层 / 数据层**。

```text
┌───────────────────────────── 感知层 ─────────────────────────────┐
│ 语音输入(STT) | 文本输入 | 设备信号 | 用户画像快照 | 外部事件流   │
└──────────────────────────────────────────────────────────────────┘
                │
┌───────────────────────────── 编排层 ─────────────────────────────┐
│ Supervisor Agent                                                   │
│ ├─ 意图路由       ├─ 风险分级       ├─ 任务编排       └─ 策略控制 │
│ 核心业务 Agent：陪伴智能体 / 健康哨兵 / 生活助手                 │
│ 支撑能力：医学RAG / 通知联络 / 摘要审计                          │
└──────────────────────────────────────────────────────────────────┘
                │
┌───────────────────────────── 执行层 ─────────────────────────────┐
│ 工具适配器：知识检索、提醒调度、消息通知、电话/短信、日志审计等   │
│ 沙箱运行时：工具权限控制、超时熔断、幂等与补偿                    │
└──────────────────────────────────────────────────────────────────┘
                │
┌───────────────────────────── 数据层 ─────────────────────────────┐
│ Redis(短期记忆/会话态) | PostgreSQL+pgvector(长期语义记忆)       │
│ Structured Facts(结构化事实) | 本地灾备存储(离线兜底)             │
└──────────────────────────────────────────────────────────────────┘
```

### 3.1 感知层（Perception Layer）

职责：统一接入多模态输入并完成标准化。

1. 语音与文本归一：STT 文本、输入时间戳、置信度、说话轮次统一建模。
2. 语义预处理：去噪、方言口音纠错、句子切分、风险关键词初筛。
3. 会话上下文封装：将用户身份、最近对话摘要、提醒状态一并注入编排层。

### 3.2 编排层（Orchestration Layer）

职责：统一决策中心，控制 Agent 执行顺序与工具调用。

1. 以 Supervisor Agent 为入口，避免多 Agent 并发失控。
2. 编排机制可基于 `Spring AI Advisors` 链或类 `LangGraph` 状态机实现。
3. 使用策略路由（规则 + 模型）决定调用哪个核心业务 Agent。
4. 对高风险场景优先抢占，普通闲聊任务自动降优先级。

### 3.3 执行层（Execution Layer）

职责：将编排结果映射为可执行动作，并保障安全与可靠性。

1. 统一工具协议（Tool Contract）：输入、输出、错误码、重试策略。
2. 工具沙箱：网络、文件、命令、数据访问按策略隔离。
3. 失败治理：超时、熔断、降级回复、人工/家属升级。

### 3.4 数据层（Data Layer）

职责：支撑记忆、检索、审计、灾备恢复。

1. 多级记忆：短期上下文 + 长期语义 + 可验证事实。
2. 事件存证：风险事件、提醒执行、关键决策链路全量记录。
3. 灾备机制：本地可恢复数据用于离线/断网情况下的最小能力保活。

## 4. Agent 体系设计：Supervisor + 核心业务 Agent

## 4.1 角色定义

### Supervisor Agent（总控）

职责：

1. 解析用户输入与上下文，进行意图分类与风险前判。
2. 选择并串联核心业务 Agent，维护单轮与多轮状态机。
3. 控制工具权限边界与调用预算（时延、token、成本）。
4. 输出最终响应及结构化执行日志。

### Companion Agent（陪伴智能体）

职责：

1. 提供日常闲聊、情绪陪伴与怀旧式话题延展。
2. 维持长期交流中的语气一致性和陪伴人格。
3. 将闲聊中识别到的可用记忆点提交给记忆治理链路。

### Health Sentry Agent（健康哨兵）

职责：

1. 对身体不适表达和风险信号进行高敏感识别与分级。
2. 在高风险场景下调用医学 RAG、预警和联络工具。
3. 输出非诊断式、可解释的风险提示与行动建议。

### Assistant Agent（生活助手）

职责：

1. 负责提醒创建、提醒确认、天气查询、联系家属等事务型任务。
2. 保障“理解 -> 执行 -> 回执”的闭环能力。
3. 在任务完成后更新对应的结构化状态与日志。

### 支撑能力（非业务主 Agent）

1. 医学 RAG：作为受控知识检索与证据约束能力，优先服务 Health Sentry。
2. 通知联络：作为工具层能力，为紧急通知和家属联络提供统一适配。
3. 摘要审计：作为平台支撑能力，负责摘要、报告、审计留痕与回放索引。

## 4.2 Tool 接入策略

各 Agent 只允许调用白名单工具，且工具按风险分级：

1. L1（低风险只读）：`getWeather`、`queryHistoricalFacts`、健康常识检索。
2. L2（中风险写操作）：`setReminder`、更新联系人备注、生成摘要。
3. L3（高风险外呼）：`notifyEmergency`、`callContact`、外部消息发送（需策略门禁）。

关键原则：

1. 不允许 Agent 直接访问底层数据库写接口，统一经服务网关。
2. `Health Sentry` 默认绑定医学检索与风险预警工具，`Assistant` 默认绑定提醒与事务工具。
3. 高风险工具必须双重条件：风险等级命中 + 状态机节点允许。
4. 所有工具调用都具备 `trace_id` 与可审计参数快照。

## 5. 多级记忆系统设计

目标：在“实时响应、长期个性化、事实准确、灾难恢复”之间取得平衡。

## 5.1 记忆分层

### Redis（短期记忆）

用途：

1. 最近 N 轮会话上下文（例如 20 轮）。
2. 临时会话状态（当前任务、确认倒计时、风险观察窗口）。
3. 限流与幂等键（防重复通知、防重复提醒）。

特性：

1. 低延迟读写，TTL 控制，适合实时链路。
2. 失效后可由长期层回填核心会话摘要。

### PostgreSQL + pgvector（长期语义记忆）

用途：

1. 长期陪伴历史、行为偏好、情绪趋势。
2. 语义向量检索（相似对话、既往风险模式、偏好召回）。

特性：

1. 结构化字段 + 向量字段混合查询。
2. 支持时间衰减与用户维度分区，避免历史噪声污染当前回复。

### Structured Facts（结构化事实层）

用途：

1. 存储“可验证且需高可信”的事实：过敏史、慢病、联系人、用药计划。
2. 通过事实优先策略覆盖纯语义记忆的“幻觉式推断”。

特性：

1. 严格 schema、版本化更新、来源可追溯。
2. 关键事实更新需确认流程（语音确认或家属端确认）。

### 本地灾备（Local DR）

用途：

1. 断网/中心服务不可达时保留最小能力。
2. 缓存最近风险词典、应急播报模板、紧急联系人快照。

特性：

1. 只读优先，恢复后异步回传审计日志与事件流水。
2. 可使用 `Kryo + 本地文件系统` 序列化最近状态快照与关键配置。
3. 数据加密存储，具备过期清理策略。

## 5.2 记忆读取优先级

建议顺序：

1. Structured Facts（高可信事实）
2. Redis（当前会话态）
3. pgvector（长期语义召回）
4. 本地灾备（仅在主链路故障时兜底）

## 6. 高风险感知与预警流程

## 6.1 风险分级策略（示例）

1. Critical：胸痛、呼吸困难、晕倒、疑似中风等立即危及生命信号。
2. High：持续剧痛、高热伴异常、明显神经系统症状等高风险信号。
3. Medium：头晕、乏力、轻度不适等需持续观察信号。
4. Low：无明确健康风险的普通表达。

## 6.2 端到端流程

```text
输入到达 → 风险初筛(规则) → 风险复核(模型) → Supervisor判级
   ├─ Low/Medium：陪伴或观察流程，增加追问与回访
   ├─ High：进入确认流程（快速二次确认 + 建议联系家属）
   └─ Critical：跳过普通对话，直接触发紧急模式与联络链路
```

### 关键机制

1. 双通道判定：规则命中保证召回，模型复核降低误报。
2. 抢占式编排：高风险任务可中断当前闲聊任务。
3. 多步确认：对可疑误触发场景提供“我没事”回退按钮。
4. 超时升级：确认超时自动升级通知家属并记录事件。
5. 全程留痕：保留触发词、判级依据、动作执行、响应结果。

## 7. 医学 RAG 优化方案

目标：在医疗相关回答中实现“可检索、可解释、可降级”，避免不当医疗建议。

## 7.1 检索架构

1. 数据源分层：指南文献 > 权威科普 > 平台维护知识库。
2. 混合检索：关键词 BM25 + 向量检索联合召回。
3. 元数据过滤：按疾病类别、年龄段、时效性过滤候选文档。

## 7.2 生成优化

1. 证据约束生成：回答必须绑定检索证据片段。
2. 风险声明模板：明确“非诊断建议”，引导线下就医。
3. 冲突消解：多证据冲突时采用“高权威优先 + 保守建议”策略。
4. 低置信度降级：检索置信不足时不强答，改为安全引导回复。

## 7.3 评估与闭环

1. 离线评估：准确率、证据覆盖率、幻觉率、风险漏检率。
2. 在线评估：响应时延、用户追问率、人工复核通过率。
3. 反馈回流：高风险问答进入标注池，驱动词典和策略持续更新。

## 8. 工程化与可观测性

## 8.1 工程化基线

1. 服务边界清晰：编排服务、工具网关、数据服务、通知服务解耦。
2. 配置中心化：模型参数、风险阈值、工具白名单统一配置化。
3. 灰度发布：按用户分组灰度，支持快速回滚。
4. 自动化测试：单元测试、契约测试、回归测试覆盖核心链路。

## 8.2 可观测性体系

1. Metrics：请求量、时延、风险触发率、工具失败率、升级率。
2. Logs：结构化日志，记录 Agent 决策与工具输入输出摘要。
3. Traces：跨层链路追踪（从感知到外呼通知）。
4. SLO/SLA：定义对话成功率、风险响应时间、通知送达率目标。

## 8.3 运行保障

1. 过载保护：并发限流、队列削峰、服务熔断。
2. 故障演练：定期演练 Redis 故障、外部通知失败、模型不可用。
3. 事件复盘：高风险事件形成标准化复盘模板，闭环到策略系统。

---

## 9. 安全沙箱与合规控制

## 9.1 安全沙箱原则

1. 最小权限：Agent 默认无权限，按需授予工具访问能力。
2. 执行隔离：高风险工具在独立运行域，限制网络与系统调用。
3. 数据最小化：只向 Agent 暴露完成任务所需最小字段。
4. 明确拒绝：对越权调用、敏感字段请求直接拒绝并告警。

## 9.2 数据安全与隐私

1. 敏感字段加密：联系人、健康档案、事件详情分级加密。
2. 传输保护：全链路 TLS，内部服务间双向认证。
3. 审计可追溯：高风险操作必须留痕，满足事后追责要求。
4. 生命周期治理：数据留存期限、脱敏导出、删除流程可执行。

## 9.3 策略管理与人工接管

1. 安全策略可配置，支持按地区与人群差异化策略。
2. 建立人机协同处置机制，预留家属或人工服务接管接口。

## 10. 分阶段路线图（目标架构落地计划）

> 说明：以下为建议节奏，表示“目标演进路径”，不代表当前仓库已完成。

## Phase 1：持久化与记忆分层

1. 将会话与关键业务数据迁移到 `PostgreSQL + Redis`。
2. 建立 Redis 短期记忆、pgvector 长期记忆、Structured Facts 事实层的统一读写抽象。
3. 打通基础审计链路，为后续风险回溯与多 Agent 编排提供数据底座。

验收重点：状态可跨会话保留，记忆召回可解释，关键事件可追踪。

## Phase 2：工具调用与闭环动作

1. 建立统一 Tool Contract，接入提醒、通知、摘要、家属联络等核心工具。
2. 通过 Function Calling / Tool Calling 让系统完成“理解 -> 执行 -> 回执”的动作闭环。
3. 将高风险场景中的通知、确认、事件记录纳入统一执行日志。

验收重点：动作执行链路可回放，提醒与预警闭环稳定，工具失败可降级。

## Phase 3：Supervisor 多 Agent 分流

1. 引入 Supervisor Agent，统一承担路由、优先级控制、冲突协调。
2. 拆分陪伴智能体、健康哨兵、生活助手，并补齐医学 RAG 与通知联络等支撑能力。
3. 通过策略编排让高风险任务可抢占普通闲聊任务，形成稳定协同模型。

验收重点：多 Agent 职责清晰，路由可解释，高风险链路优先级正确。

## Phase 4：灾备与极端环境鲁棒性

1. 引入本地灾备存储，保留关键联系人、应急模板、最近状态快照。
2. 建立断网、依赖异常、服务重启场景下的最小可用能力与恢复机制。
3. 完成本地兜底与中心链路恢复后的异步回传和审计对账。

验收重点：极端环境下仍可执行核心求助与提醒能力，恢复过程可追踪。

---

## 11. 面试与评审视角下的架构亮点

1. 以 Supervisor 统一决策，避免多 Agent 竞争导致不可控行为。
2. 采用“语义记忆 + 事实记忆”双轨，兼顾个性化与准确性。
3. 风险链路具备“召回优先 + 误报抑制 + 超时升级”完整闭环。
4. 将可观测性和安全沙箱前置到架构层，而非事后补丁。
5. 通过分阶段路线图，将愿景拆解为可交付、可验收里程碑。

## 12. 待决策项（实施前需确认）

1. 医学知识源清单与更新频率（权威来源治理机制）。
2. 紧急外呼策略在不同地区的合规边界与默认动作。
3. 本地灾备的数据介质、容量上限、过期策略。

## 13. 结论

GoldenAge Guardian 是“老人陪伴助手”的下一代目标架构：  
它并非当前实现现状，而是面向未来 6-12 个月的工程化演进蓝图。  
该架构以安全、可靠、可解释为核心，兼顾陪伴体验与风险处置能力，支持产品从功能可用走向长期可运营、可审计、可规模化。
