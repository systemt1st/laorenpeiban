# 老人陪伴助手 产品需求文档（PRD）

文档版本：v2.0（重写版）
文档日期：2026-03-30
适用范围：产品、设计、研发、测试、运营、合规

---

## 1. 文档定位与命名约定

### 1.1 对外产品名
对外统一产品名：**老人陪伴助手**。

### 1.2 内部架构代号
`GoldenAge Guardian（金色年华守护者）` 为内部下一代架构代号，仅用于研发与架构演进讨论，不作为对外品牌名称。

### 1.3 版本语义
- **v1 基线（当前实现）**：当前代码库可支持的核心能力集合。
- **v2 目标态（演进方向）**：面向下一阶段架构与体验升级的目标，不代表已全部落地。

---

## 2. 产品愿景与范围

### 2.1 产品愿景
面向 60 岁及以上用户，提供低门槛、可持续、可信赖的智能陪伴服务，兼顾情感陪伴、健康提醒与紧急风险应对能力，降低独居与半独居老人日常生活风险。

### 2.2 核心价值主张
- 降低孤独感：通过自然对话形成稳定陪伴关系。
- 降低执行成本：通过语音优先提醒机制提升日常健康任务完成率。
- 降低突发风险：通过分级识别与应急流程提高关键事件响应速度。
- 降低家属焦虑：通过可追溯事件摘要与授权共享提升家庭协同效率。

### 2.3 产品边界
- 本产品不是医疗诊断系统，不输出诊断结论。
- 本产品提供健康建议与应急引导，不替代医生、120 与线下救助体系。
- 本产品默认服务家庭养老场景，机构养老能力为后续扩展方向。

## 3. 现状与目标：v1 基线 vs v2 演进

### 3.1 v1 基线（当前实现）

已具备的核心能力：
- 语音/文本对话入口，支持基础多轮沟通。
- 日常提醒（吃药、喝水、复诊、自定义）与确认机制。
- 紧急关键词识别、分级触发、紧急确认与拨号引导。
- 基础用户资料、健康档案、紧急联系人管理。
- 家属侧摘要查看与事件追溯能力。
- 适老化界面基础能力（大字号、高对比、关键按钮放大）。

当前限制（需在 v2 解决）：
- 记忆能力偏会话级，跨会话连续性有限。
- 画像更新机制偏静态，长期行为变化感知不足。
- 紧急识别主要依赖关键词，语义理解与上下文推断能力有限。
- 家属协同偏“看结果”，缺少主动协作编排能力。

### 3.2 v2 目标态（演进方向）

`GoldenAge Guardian` 目标是形成“情感陪伴 + 健康管理 + 风险治理 + 家庭协同”的统一智能体架构。

目标特征：
- 以用户长期记忆为核心，支持跨天、跨周、跨场景连续理解。
- 在不增加老人操作负担的前提下，提升提醒执行闭环与风险前置发现能力。
- 构建可解释、可审计、可回溯的安全与合规能力。
- 支持多终端、多角色协同（老人、家属、后续可扩展护理角色）。

说明：以上为演进方向，不代表当前版本已完整实现。

### 3.3 非目标（当前阶段不承诺）
- 不承诺替代医疗诊断与急救专业判断。
- 不承诺在弱网/断网场景下提供完整云端智能能力。
- 不承诺一次性接入所有硬件生态与全部区域医疗系统。

## 4. 用户画像与核心场景

### 4.1 画像 A：独居/半独居老人
- 年龄：65-85 岁
- 数字能力：可完成基础语音或简单触屏操作
- 核心诉求：有人说话、记得做事、出事能求助
- 关键痛点：忘记吃药、夜间孤独、突发不适时慌乱

### 4.2 画像 B：关注型家属
- 年龄：30-55 岁
- 角色：子女/主要照护者
- 核心诉求：了解老人状态、及时感知风险、减少信息不对称
- 关键痛点：异地照护难、无法实时判断风险等级、担心漏看异常

### 4.3 高频场景
- 日常闲聊：早晚问候、回忆聊天、情绪疏导
- 健康任务：吃药、喝水、复诊、睡眠与活动提醒
- 突发事件：胸闷、摔倒、头晕等表达触发应急流程
- 家庭协同：家属查看摘要、接收预警、调整提醒策略

## 5. 四大能力域需求

### 5.1 能力域一：闲聊陪伴

v1 基线：
- 支持日常闲聊、问候、基础信息问答。
- 支持语音播报，降低阅读负担。

v2 演进方向：
- 建立“陪伴人格一致性”，提升长期交流稳定感。
- 支持基于记忆的主动关怀（如回访未完成事项、延续历史话题）。
- 支持情绪波动识别与陪伴策略切换（安抚、分散注意、转家属协同）。

验收重点：
- 连续对话满意度提升。
- 主动关怀触发后的用户有效回应率提升。

### 5.2 能力域二：健康关注（非诊断）

v1 基线：
- 以提醒任务为主，覆盖吃药、喝水、复诊等场景。
- 支持提醒确认、二次提醒、未确认记录。

v2 演进方向：
- 从“提醒触发”升级为“任务闭环管理”（计划、执行、确认、异常解释）。
- 引入趋势感知（如连续未确认、昼夜节律异常）用于风险前置提示。
- 支持家属协同策略（提醒时段调整、优先级策略）。

验收重点：
- 提醒确认率、连续执行率、异常漏报率。

### 5.3 能力域三：紧急预警

v1 基线：
- 基于风险关键词与规则进行分级触发。
- 支持紧急确认、拨打 120/联系家属、事件记录。

v2 演进方向：
- 融合语义理解、上下文连续性与历史风险画像，提升识别准确性。
- 建立分级响应编排：快速确认、行动建议、通知链路、事后追踪。
- 提供“误报可纠正、真报可追溯”的审计能力。

验收重点：
- 高风险识别召回率。
- 平均应急响应时延。
- 误报率与漏报率。

### 5.4 能力域四：生活助记

v1 基线：
- 支持基础提醒与简单信息记录。

v2 演进方向：
- 支持“人-事-时间”结构化记忆（重要日期、用药习惯、家庭事件）。
- 支持自然语言创建与修改助记事项。
- 支持对老人友好的“今日要事”复述与确认。

验收重点：
- 助记事项创建成功率。
- 助记事项按时完成率。

## 6. 紧急风险分级（保留并升级）

说明：本章描述的是统一分级框架。当前 v1 以关键词与规则触发为主，v2 才逐步引入上下文、记忆和历史画像增强。

### 6.1 分级标准
- **极高风险（critical）**：疑似危及生命征兆，如胸痛、呼吸困难、晕倒、大出血、中风疑似。
- **高风险（high）**：可能快速恶化征兆，如剧痛、持续高烧、严重头晕、明显出血。
- **中风险（medium）**：一般不适征兆，如乏力、恶心、轻度头痛、腹部不适。

### 6.2 分级响应原则
- critical：默认立即进入紧急模式并引导拨打 120，同时通知家属。
- high：快速二次确认，确认后进入紧急流程；未确认时保持高频追问。
- medium：优先安抚与观察，必要时建议联系家属并持续监测。

### 6.3 设计要求
- v1 当前实现以规则与关键词优先，保证高风险表达的基础召回。
- v2 目标态在此基础上叠加上下文、历史状态与结构化画像，降低误报与漏报。
- 所有关键决策必须有事件日志，支持追溯与复盘。
- 紧急场景下对话范围收敛，优先输出行动指令而非开放闲聊。

## 7. 关键业务流程

说明：本章同时包含当前可运行流程与目标增强流程。各小节标题中的标注用于区分“当前已具备”与“后续增强”。

### 7.1 首次建档流程（v1 当前）
进入应用 -> 语音引导建档 -> 填写称呼与基础信息 -> 设置紧急联系人（至少 1 位） -> 选择提醒模板 -> 完成

### 7.2 日常陪伴流程（v1 当前 + v2 增强）
v1 当前：用户发起语音/文本 -> 系统完成基础意图识别 -> 输出陪伴回复 -> 记录会话内容
v2 增强：在 v1 基础上追加记忆与画像召回 -> 延续历史话题 -> 记录结构化关键记忆点

### 7.3 提醒闭环流程（v1 当前 + v2 增强）
v1 当前：任务到时触发 -> 语音播报 + 大按钮确认 -> 未确认二次触发 -> 仍未确认则记录异常
v2 增强：在 v1 基础上按策略通知家属 -> 家属回写提醒策略 -> 系统同步后续提醒节奏

### 7.4 紧急预警流程（v1 当前 + v2 增强）
v1 当前：风险表达触发 -> 关键词/规则分级 -> 执行确认、拨号或联系家属 -> 事件沉淀
v2 增强：在 v1 基础上叠加上下文、记忆和画像 -> 优化分级与响应编排 -> 完成后续追踪与复盘

### 7.5 家庭协同流程（v1 当前 + v2 增强）
v1 当前：家属授权登录 -> 查看摘要与事件
v2 增强：在 v1 基础上调整提醒/联系人策略 -> 系统同步到老人侧执行

## 8. 记忆与画像需求

### 8.1 画像层级
- 基础画像：年龄段、称呼、交互偏好、可访问性偏好。
- 健康画像：慢病、用药计划、过敏信息、就诊计划。
- 社会画像：常联系家属、重要关系与纪念日。
- 行为画像：作息规律、提醒确认习惯、常见情绪波动。

### 8.2 记忆分层
- 会话记忆：当前对话上下文，保证回复连贯。
- 短期记忆：近 7-30 天重点事项，用于提醒和回访。
- 长期记忆：稳定偏好与关键人生信息，用于长期陪伴个性化。

### 8.3 记忆治理要求
- 记忆写入：仅写入与服务目标强相关的信息，避免过度采集。
- 记忆纠错：支持用户/家属主动更正。
- 记忆遗忘：按策略衰减低价值信息，降低冗余与隐私风险。
- 记忆可见：关键画像与记忆支持用户可查看、可导出、可删除。

## 9. 安全与合规要求

### 9.1 隐私与数据保护
- 最小必要采集：仅采集实现服务所需数据。
- 传输与存储加密：敏感字段脱敏与分级保护。
- 生命周期管理：设定保留期限与自动清理策略。

### 9.2 合规原则
- 首次使用需明确告知并取得授权同意。
- 支持个人信息查询、更正、导出、删除。
- 遵循《个人信息保护法》等适用法规。

### 9.3 医疗与安全边界
- 明示“非医疗诊断”边界。
- 高风险场景优先引导线下医疗与急救资源。
- 输出内容需经过安全策略过滤，避免误导与危险建议。

### 9.4 审计与可追溯
- 关键风险事件全链路留痕（触发、判断、动作、通知）。
- 支持内部安全审计与异常复盘。

## 10. 技术实现约束（非绑定）

- PRD 定义产品能力与质量目标，不绑定特定后端语言、数据库或单一云厂商。
- 当前 v1 存在参考实现，但 v2 可按目标能力进行技术替换与架构升级。
- 技术选型应满足：高可用、可扩展、可审计、低延迟、低运维复杂度。

## 11. 核心指标（产品与运营）

### 11.1 体验指标
- 日均有效对话轮次
- 陪伴满意度（主观反馈）
- 语音交互成功率

### 11.2 健康与助记指标
- 提醒确认率
- 助记事项完成率
- 连续 7 天执行达成率

### 11.3 风险治理指标
- 紧急事件识别召回率
- 紧急流程平均响应时延
- 误报率/漏报率

## 12. 路线图（Phase 1-4）

### Phase 1：持久化与记忆分层（近期）
目标：从单体会话应用升级为具备持续状态管理能力的系统基座。
重点：
- 迁移到 `PostgreSQL + Redis`，补齐对话持久化和会话热数据管理。
- 建立短期记忆、长期记忆、结构化事实的分层抽象。
- 为后续风险回溯、主动关怀和多 Agent 编排预留统一数据接口。

### Phase 2：工具调用与闭环动作（中期）
目标：让系统具备“理解之后能执行”的闭环能力。
重点：
- 引入 Function Calling / Tool Calling 机制。
- 落地提醒创建、提醒确认、风险预警、通知家属等关键工具。
- 建立可追踪的动作执行日志，保证提醒和预警流程可回放。

### Phase 3：Supervisor 多 Agent 分流（中长期）
目标：从单一对话 Agent 升级为多角色协同编排系统。
重点：
- 引入 Supervisor 统一负责任务分发、冲突协调和优先级控制。
- 拆分陪伴智能体、健康哨兵、生活助手等核心业务 Agent。
- 让闲聊、提醒、风险识别和通知链路从“串在一起”转为“按职责编排”。

### Phase 4：灾备与极端环境鲁棒性（长期）
目标：在弱网、抖动和局部依赖异常下保持最低可用能力。
重点：
- 引入本地灾备与离线恢复机制，保障重启后快速恢复关键状态。
- 补齐高风险词典、关键联系人、应急播报模板等本地兜底能力。
- 提升系统在养老真实场景中的连续可用性与安全性。

说明：以上路线图描述的是演进计划，不代表当前版本已经完成。

## 13. 风险与依赖

- 语音识别准确率受设备、环境噪声和口音影响。
- 紧急事件识别对误报和漏报容忍度低，需要持续调优。
- 合规要求存在地域与政策差异，需要持续法务校验。
- 家属协同效果依赖家庭参与度与通知触达成功率。

---

## 14. 结语

本 PRD 作为“老人陪伴助手”新版产品基准文档，明确了当前 v1 基线、v2 演进方向及 `GoldenAge Guardian` 的内部架构定位。后续迭代以“能力可验证、风险可追溯、体验可持续”为实施原则推进，不将目标态默认视为已落地能力。
